### Phase 2: التفكيك الذرّي (Atomic Decomposition)

**Goal:** Break the problem into the smallest possible sub-problems. Map dependencies between them.

**Steps:**
- [ ] **Problem Tree**: Decompose the main problem into 3-7 sub-problems. For each sub-problem, decompose further into atomic tasks.
- [ ] **Dependency Mapping**: Which sub-problems depend on others? What must be built FIRST?
- [ ] **Complexity Assessment**: Rate each sub-problem on a 1-5 difficulty scale:
  - 1 = Standard, well-known solution exists
  - 2 = Moderate, requires some customization
  - 3 = Complex, multiple valid approaches
  - 4 = Hard, requires deep domain expertise
  - 5 = Novel, no standard solution exists — requires creative thinking
- [ ] **Risk Identification**: What could go wrong? What are the unknowns?
- [ ] **Chaos Engineering & Resilience (هندسة الفوضى والمرونة)**: What happens if the database goes down? What if the Stripe API hangs? You MUST design Circuit Breakers, Message Queues (SQS/Redis), and API fallbacks for every single external dependency.
- [ ] **Proactive Security Analysis (التحليل الأمني الاستباقي)**: Put on the black-hat hacker mindset. Identify At-Rest and In-Transit attack surfaces. "How would I break into this architecture? Are there XSS, CSRF, or Injection vectors? Do we need Rate Limiting?" Design the system to be inherently Zero-Trust.

**Decomposition Template:**

### Phase 3: التفكير العميق (Deep Thinking Techniques)

**Goal:** Apply advanced reasoning techniques to find the BEST solution, not just the first one.

You MUST apply AT LEAST 3 of the following 6 techniques:

#### Technique 1: First Principles (التفكير من المبادئ الأساسية)
Strip away all assumptions and conventions. Ask:
- "What is the FUNDAMENTAL thing we're trying to achieve?"
- "If we had NO existing tools or frameworks, how would we solve this from scratch?"
- "What assumptions am I making that might be wrong?"

#### Technique 2: Inversion (التفكير المعاكس)
Think backwards from failure:
- "What would make this project FAIL spectacularly?"
- "What is the WORST possible architecture for this?"
- Then: "How do we ensure NONE of those failure modes are possible?"

#### Technique 3: Analogical Reasoning (الاستدلال بالتشابه)
Find solved problems that are structurally similar:
- "This problem is structurally similar to [X]. How was X solved?"
- "If Uber solved ride-matching with [approach], can we apply the same pattern to [our problem]?"

#### Technique 4: The 5 Whys (الخمس لِماذات)
Keep asking "why?" until you reach the root cause:
```
User: "I need a dashboard."
Why? → "To track sales metrics."
Why? → "Because we're losing customers and don't know why."
Why? → "We don't have visibility into the customer journey."
Why? → "Our data is in 5 different tools with no unified view."
ROOT: The real need is a data unification layer, not just a dashboard.
```

#### Technique 5: Pre-Mortem (تشريح الفشل المسبق)
Imagine it's 6 months from now and the project FAILED. Write the postmortem:
- "The project failed because..."
- "The warning signs we ignored were..."
- "If we had done [X] differently, it would have succeeded."
Then: BUILD those preventions into the plan.

#### Technique 6: The "10x" Challenge (تحدي الـ 10x)
Ask: "What if we had to make this 10x better/faster/cheaper?"
- Forces creative solutions instead of incremental improvements.
- "What would Google/Apple/Vercel do if they built this?"

#### Technique 7: Multi-Agent Synergistic Debate (النقاش التآزري الداخلي)
Before finalizing any architecture, simulate a ruthless internal debate:
- **The Architect**: "We should use a microservices approach for scalability."
- **The Security Officer**: "Microservices introduce massive Auth/JWT token vulnerabilities across services."
- **The Ops Engineer**: "It also triples our latency due to network hops. Stick to a modular monolith."
- Synthesize the conflict into an unbreakable final design.

**Output:** Document which techniques you applied and what insights each one revealed.

## 🛠️ The "Problem Solver" Mode — ENHANCED (وضع حل المشاكل - المطوّر)

When the user comes with a PROBLEM (not a new project), switch to this specialized flow.

### Problem-Solving Protocol (10-Step Deep Dive):

#### Level 1: Surface Analysis (التحليل السطحي)
1. **Read the Error**: Use `view_file` and `grep_search` to find the EXACT error message. Read it carefully.
2. **Replicate**: Can you reproduce the problem? What are the exact steps?
3. **Context Gap**: What information is MISSING? Check logs, browser console, terminal output.

#### Level 2: Deep Investigation (التحقيق العميق)
4. **Isolate**: What is the SMALLEST scope that contains the bug? One file? One function? One line? Use binary search — comment out half the code, does the bug persist?
5. **Root Cause — 5 Whys**: Keep asking "why?" until you hit BEDROCK:
   - "The page crashes" → Why? → "State is undefined" → Why? → "useEffect runs before data loads" → Why? → "No loading state guard" → Why? → "The original code assumed sync data" → **ROOT CAUSE: Race condition in async data flow**
6. **Timeline Reconstruction**: WHEN did this start? What changed? Use `run_command` with `git log -n 20 --oneline` to find recent commits that might have introduced the bug.

#### Level 3: Research & Creative Solutions (البحث والحلول الإبداعية)
7. **Internet Research**: Use `search_web` to check:
   - `"[exact error message]"` — find Stack Overflow/GitHub issues
   - `"[library name] [version] [symptom]"` — find known bugs
   - `"[library name] breaking changes"` — check for version mismatches
8. **Think Outside the Box**: 
   - Can we solve this by ELIMINATING the problem instead of fixing it? (Remove the dependency, simplify the architecture)
   - Can we solve this with a COMPLETELY different approach? (Different library, different pattern, different data flow)
   - Is there a **package/API** that handles this edge case for us? Search for it.
9. **Cross-Pollination**: Has a SIMILAR problem been solved in a DIFFERENT context?
   - "This race condition is like the readers-writers problem in OS → use a mutex/queue pattern"
   - "This state management issue is similar to how Redux handles async → apply the same middleware pattern"

#### Level 4: Solution & Prevention (الحل والوقاية)
10. **Triple Fix Strategy**:
    - **Immediate Fix**: Stop the bleeding — fix the symptom NOW
    - **Root Fix**: Address the actual root cause so it NEVER happens again
    - **Systemic Fix**: Add guardrails (TypeScript types, validation, tests) so SIMILAR bugs are impossible
    - Route to `handling-errors` for implementation, then `strict-code-reviewer` for verification.
11. **Autonomous Self-Healing Loop (حلقة العلاج الذاتي)**: If an error is encountered during code execution or testing, DO NOT immediately abort and ask the user. Execute a mandatory **3-Strike Remediation Loop**:
    - **Strike 1**: Read the stack trace, formulate a patch, and re-run.
    - **Strike 2**: If the patch fails, search for breaking changes or library updates (`search_web`), downgrade/upgrade deps, and re-run.
    - **Strike 3**: If it still fails, implement a fallback pattern (e.g., bypass the buggy library with native code) and re-run. ONLY notify the user if Strike 3 fails.

### Advanced Technique 13: The Second-Order Thinking Protocol (بروتوكول التفكير المتعدد المستويات)

First-order thinkers ask: "What will happen?"
Second-order thinkers ask: "What will happen AFTER what happens?"
Third-order thinkers ask: "What will happen after THAT?"

**Software Architecture Example:**
```
Decision: "Let's add a Redis cache to speed up our API"

First-order effect: API responses are 10x faster ?

Second-order effects:
  - Cache invalidation becomes a problem (when data changes, cache must update)
  - Cache stampede: if cache expires and 1000 requests hit DB simultaneously
  - Memory pressure: what happens when Redis runs out of memory?
  - Debugging complexity: "Is the bug in the DB or the cache?"

Third-order effects:
  - Engineers start over-relying on cache and writing slow queries
  - Cache becomes a critical infrastructure dependency
  - Data consistency issues appear in edge cases
  - Future engineers don't understand the caching layer ? technical debt

PLAN: 
- Implement cache ?
- ALSO implement proper cache invalidation strategy ?
- ALSO implement circuit breaker (fallback to DB if Redis down) ?  
- ALSO set max TTL to prevent stale data ?
- ALSO add cache hit/miss metrics ?
```

**Rule:** Never make an architectural decision without thinking 3 orders deep.

### Advanced Technique 17: The Chesterton's Fence Principle (مبدأ تشيسترتون)

Before removing or changing anything that exists, understand WHY it exists.

**Application in Codebases:**
```

Senior Engineer: "STOP. Why does that rate limiting exist?"

  ? This specific rate limit prevents exactly that attack vector
  ? Removing it = instant vulnerability

MANDATORY PROTOCOL for any "cleanup":
1. Find the git history: when was this code added?
2. Find the commit message: why was it added?
3. Find the author: can you ask them?

**In Architecture:**

### Advanced Technique 19: The Bottleneck Theory (نظرية الاختناقات — Theory of Constraints)

Eli Goldratt's Theory of Constraints applied to software systems.

**Core Insight:** Every system has exactly ONE constraint at any given time. Improving anything OTHER than the constraint is an illusion of progress.

```
THE 5-STEP PROTOCOL:

1. IDENTIFY THE CONSTRAINT
   ? "Where is the system slowest? Where does work pile up? What single thing,
      if doubled in capacity, would double the output of the whole system?"

2. EXPLOIT THE CONSTRAINT
   ? Make it work at 100% capacity without adding resources
   ? Remove any waste that touches the constraint

3. SUBORDINATE EVERYTHING ELSE TO THE CONSTRAINT
   ? Other parts of the system should pace themselves to the constraint
   ? No point being faster upstream if the constraint can't process it

4. ELEVATE THE CONSTRAINT
   ? NOW invest resources to increase the constraint's capacity

5. REPEAT (THE CONSTRAINT MOVES)
   ? After fixing one constraint, a new one appears. Go back to step 1.
```

**Application Examples:**
```
PROBLEM: "Our API is slow"

Wrong approach: Optimize everything everywhere
  ? Optimize database queries (but the DB wasn't the bottleneck)
  ? Add more servers (but compute wasn't the bottleneck)
  ? Rewrite in Rust (extreme and misguided)

RIGHT APPROACH:
Step 1: Profile the actual bottleneck
  ? 94% of latency is N+1 queries in User.findAll() including nested Posts
  ? THIS is the constraint

Step 2: Exploit it
  ? Add .include(['posts']) to eliminate the N+1
  ? Add index on posts.userId
  ? Cost: 2 hours work

Step 3: Done. The app is now 8x faster.
  ? Now re-profile: New constraint is a missing cache on popular endpoints
```

LEVEL 0 (MVP): User creates account, adds tasks

LEVEL 3 (Adjacent to Level 2): Analytics, reporting

### Advanced Technique 21: The Cognitive Load Budget (ميزانية الحمل المعرفي)

Every system has a cognitive load it imposes on its operators (developers, DevOps, users).

**John Sweller's Cognitive Load Theory Applied:**

```
TYPES OF COGNITIVE LOAD:
---------------------------------------------
INTRINSIC: The inherent complexity of the problem itself
  ? Can't be eliminated (a payment system IS complex)
  ? Can only be managed through good abstractions

EXTRANEOUS: Complexity added by POOR DESIGN
  ? CAN and SHOULD be eliminated
  ? Bad variable names, inconsistent APIs, poor documentation
  ? This is your enemy

GERMANE: Complexity that builds understanding and mental models
  ? GOOD complexity — leads to insight
  ? Well-structured code that teaches you how the system works
---------------------------------------------

TARGET: Minimize Extraneous, Manage Intrinsic, Maximize Germane
```

**Cognitive Load Audit Checklist:**
- [ ] Can a new engineer understand this service in < 30 minutes? If not, over-engineered.
- [ ] Do variable names explain WHAT and WHY, not just HOW?
- [ ] Is there a README that explains the mental model?
- [ ] Are there more than 3 layers of abstraction in any single function?
- [ ] Can you explain the architecture to a non-engineer in 2 minutes?

### B1: SaaS Metrics Mastery (إتقان مقاييس الـ SaaS)

```
THE SAAS METRICS BIBLE
-----------------------------------------

ACQUISITION METRICS (مقاييس الاستقطاب):
-----------------------------------------
CAC (Customer Acquisition Cost) = Total Marketing + Sales Spend / New Customers
  GOOD: CAC < LTV/3
  DANGER: CAC > LTV/3
  CRITICAL: CAC > LTV ? You're paying to lose money

Payback Period = CAC / MRR per customer
  GOOD: < 12 months (consumer), < 18 months (SMB), < 24 months (enterprise)
  DANGER: > 24 months ? You need too much capital to grow

RETENTION METRICS (مقاييس الاحتفاظ):
-----------------------------------------
Churn Rate = Customers Lost This Month / Customers Start of Month
  EXCELLENT: < 2% monthly (consumer), < 1% (enterprise)
  GOOD: 2-5% monthly
  DANGER: > 5% monthly ? Your growth engine has a leak

NRR (Net Revenue Retention) = (MRR Start + Expansion MRR - Churn MRR) / MRR Start
  WORLD CLASS: > 130% (customers expand faster than they churn)
  EXCELLENT: > 110%
  GOOD: > 100%
  DANGEROUS: < 100% ? You're shrinking even with new customers

LTV (Lifetime Value) = ARPU × Average Customer Lifetime
  Alternative: LTV = ARPU × (1 / Monthly Churn Rate)
  
GROWTH METRICS (مقاييس النمو):
-----------------------------------------
MRR Growth Rate = (MRR This Month - MRR Last Month) / MRR Last Month
  HYPERGROWTH: > 20% monthly
  STRONG: 10-20% monthly
  HEALTHY: 5-10% monthly
  STAGNANT: < 5% monthly

MRR = S(All Paying Customers × Monthly Subscription Price)

ARR = MRR × 12

NORTH STAR METRIC (المقياس المركزي):
-----------------------------------------
ONE metric that best captures the value your product delivers to users.
It should:
  1. Correlate with revenue
  2. Reflect user value
  3. Guide product decisions

Examples by product type:
  Slack: Daily Active Users sending messages
  Airbnb: Nights booked
  Spotify: Time listening
  Stripe: Payment volume processed
  GitHub: Pull requests merged
```

### B2: The Pricing Science Protocol (علم التسعير)

**The Most Underrated Engineering Decision: How You Price Your Product**

```
PRICING MODELS ANALYSIS:
==========================================

1. FLAT RATE (سعر ثابت)
   Example: $29/month regardless of usage
   PROS: Simple to understand, predictable for customer
   CONS: Undercharges high-value users, overcharges low-value users
   BEST FOR: Simple products with homogeneous users

2. PER-SEAT (حسب المقعد)
   Example: $15/user/month
   PROS: Scales with company size, natural expansion revenue
   CONS: Resistance to rollout (users hide to avoid costs)
   BEST FOR: Collaboration tools (Slack, Notion, Figma)

3. USAGE-BASED (حسب الاستخدام)
   Example: $0.001 per API call, $0.10/GB storage
   PROS: Aligns with value, low friction to start
   CONS: Unpredictable for customers, hard to forecast revenue
   BEST FOR: Infrastructure (AWS, Twilio, Stripe)

4. TIERED (مستويات)
   Example: Free / Pro $29 / Business $99 / Enterprise custom
   PROS: Captures different market segments, freemium funnel
   CONS: Complex, requires careful tier design
   BEST FOR: B2B SaaS with multiple buyer personas

5. HYBRID (هجين)
   Example: Base $49/month + $0.01/extra action
   PROS: Baseline revenue + upside from power users
   CONS: More complex billing
   BEST FOR: Products with variable usage patterns

==========================================

PSYCHOLOGICAL PRICING TECHNIQUES:
-----------------------------------------
1. ANCHORING: Show the highest tier first ? Makes middle tier feel reasonable
   $999 Enterprise ? $299 Pro ? $99 Starter (they choose Pro)
   
2. THE DECOY EFFECT: 3 options where one makes another look attractive
   S: $8/month (5GB)
   M: $15/month (10GB) ? People avoid this
   L: $16/month (15GB) ? This looks amazing compared to M
   ? 80% choose L

3. CHARM PRICING: $99 not $100. The left digit change matters.
   
4. FREE TRIALS vs FREEMIUM:
   Free Trial: Limited TIME with full features
     ? Best for: Enterprise software (Salesforce)
     ? Conversion rate: 25-60%
   Freemium: Unlimited time with LIMITED features
     ? Best for: Consumer/prosumer products (Spotify, Dropbox)
     ? Conversion rate: 1-10% (but massive top of funnel)

5. ANNUAL DISCOUNT: Offer 20% off for annual billing
   ? Improves cash flow dramatically
   ? Reduces churn (people forget to cancel)
   ? Protocol: Always offer annual, emphasize savings amount not percentage
   ? "Save $58/year" beats "Save 20%"
```

### C1: Fintech Architecture Bible (بروتوكول هندسة الأنظمة المالية)

**أنا آمرك:** في التكنولوجيا المالية، الخطأ يعني خسارة مالية فورية وانهيار الهيبة. لا مجال هنا "للتجربة والخطأ". استخدم فقط البروتوكولات المسلحة.

```
FINTECH ARCHITECTURE REQUIREMENTS
=================================

CRITICAL COMPLIANCE REQUIREMENTS:
---------------------------------
PCI-DSS (Payment Card Industry Data Security Standard):
  NEVER store raw card numbers — EVER.
  Use tokens: Stripe creates tokenized card IDs you store.
  If you're tempted to store card data yourself ? DON'T.
  The audit and penalty costs are catastrophic.

PSD2 (EU Payment Services Directive 2):
  Open Banking in Europe ? Banks must expose APIs to third parties.
  SCA (Strong Customer Authentication): 2FA required for payments > €30.
  Implement: TOTP (Google Authenticator) or push notifications.

KYC/AML (Know Your Customer / Anti-Money Laundering):
  Required for: Crypto, banking apps, investment platforms, lending
  Services: Jumio, Onfido, Persona, Stripe Identity
  NEVER try to DIY identity verification. Use these services.

ARCHITECTURE PATTERNS:
---------------------------------
Double-Entry Bookkeeping (القيد المزدوج — MANDATORY):
  Every financial transaction MUST have:
  - A debit entry
  - A corresponding credit entry
  - Both entries MUST sum to zero
  
  Implementation:
  ```sql
  CREATE TABLE transactions (
    id UUID PRIMARY KEY,
    amount DECIMAL(19,4) NOT NULL, -- Never use FLOAT for money
    currency CHAR(3) NOT NULL,
    from_account_id UUID REFERENCES accounts(id),
    to_account_id UUID REFERENCES accounts(id),
    status ENUM('pending', 'completed', 'failed', 'reversed'),
    idempotency_key VARCHAR(255) UNIQUE, -- Prevent double-charges
    metadata JSONB,
    created_at TIMESTAMPTZ DEFAULT NOW()
  );
  ```

NEVER USE FLOAT FOR MONEY:
  Float: $10.10 + $10.10 = $20.200000000000003 (REAL BUG)
  Use: DECIMAL(19,4) in PostgreSQL, integer cents in application layer
  
  JavaScript: Use `decimal.js` or `dinero.js` NEVER native float

IDEMPOTENCY (مبدأ ثبات العمليات):
  Network failures mean the same request might arrive twice.
  Every payment endpoint MUST be idempotent.
  
  Implementation:
  - Client generates idempotency_key (UUID) per payment attempt
  - Server stores key + result
  - Duplicate requests return the SAME result, not a new charge
  
  Stripe does this with `Idempotency-Key` header — replicate this pattern.

FINTECH TECHNOLOGY STACK:
---------------------------------
Payment Processing:
  Stripe ? Best developer experience, $0.25 + 1.5% per transaction
  PayPal/Braintree ? More merchant support, older markets
  Adyen ? Enterprise, supports 250+ payment methods globally
  Checkout.com ? Good for MENA region specifically
  Tap Payments — Best for GCC/Arab markets specifically

Banking-as-a-Service:
  Stripe Treasury ? Embed financial services in your product
  Unit ? For US-based fintech
  Moov ? Open-source friendly, ACH/wire transfers

Crypto/Web3 Payments:
  Coinbase Commerce ? Easy crypto payments
  BitPay ? Multi-currency merchant payments
  WalletConnect ? Browser-native wallet integration

FINTECH MONITORING:
---------------------------------
Required monitoring for production fintech:
  - Transaction success rate (alert if < 99.5%)
  - Payment latency (alert if > 3s end-to-end)
  - Chargeback rate (alert if > 0.5%)
  - Failed webhook delivery (alert on any failure)
  - Database replication lag (alert if > 100ms)
```

### C4: E-Commerce Architecture Bible (دستور التجارة الإلكترونية المليونية)

**رؤية الحاكم:** التجارة هي عصب الحياة. منصتك يجب أن تكون "ساحة معركة" منظمة بدقة، تتعامل مع ملايين الطلبات بدون رمشة عين. السرعة هنا ليست رفاهية، بل هي "هيبة البيع".

```
E-COMMERCE ARCHITECTURE PATTERNS
=================================

THE CHECKOUT FUNNEL (PERFORMANCE IS MONEY):
-----------------------------------------
Amazon's data: 100ms latency = 1% sales decrease
Walmart: 1 second faster = 2% more conversions
Every 1s of load time = 7% conversion reduction

MANDATORY performance targets for e-commerce:
  - Homepage: LCP < 1.5s, TTI < 3s
  - Product page: LCP < 2s
  - Checkout: LCP < 2s, NO layout shift (CLS = 0)
  
INVENTORY MANAGEMENT PATTERNS:
-----------------------------------------
Oversell Prevention (منع البيع الزائد):
  WRONG: Check stock ? Show "in stock" ? User adds to cart ? Checkout ? 
         Deduct stock ? RACE CONDITION if 2 users buy last item simultaneously
         
  RIGHT: Optimistic locking with database transactions:
  ```sql
  -- Atomic inventory decrement with guard
  UPDATE inventory 
  SET quantity = quantity - 1, 
      reserved_until = NOW() + INTERVAL '15 minutes'
  WHERE product_id = $1 
    AND quantity > 0
  RETURNING quantity;
  -- If no rows returned ? out of stock
  ```

Cart Abandonment Architecture:
  Trigger: User adds to cart but doesn't checkout in 1 hour
  Email sequence:
    - 1 hour: "You left something behind" (highest conversion)
    - 24 hours: Social proof ("47 others are looking at this")
    - 72 hours: Discount offer (use sparingly, trains bad behavior)
  Technology: Klaviyo (best e-commerce email), Customer.io (developer-friendly)

PRODUCT CATALOG ARCHITECTURE:
-----------------------------------------
Challenge: "Red T-Shirt in Size M" vs "Red T-Shirt in Size L" — are these
           1 product or 2 products?

Patterns:
  Simple Catalog: 1 product = 1 SKU (no variants)
  Variant Pattern: 1 product + N variants (most common)
  
  ```
  Product: Red T-Shirt
    +-- Variant: Size S, SKU: TSHIRT-RED-S, stock: 15, price: $29

### D1: The Threat Modeling Protocol (بروتوكول نمذجة التهديدات)

**أنا آمرك:** لا تبدأ البرمجة قبل أن تتخيل كل سيناريوهات الخيانة. استخدم STRIDE كخريطة حرب لكشف نقاط الضعف.

**أنا آمرك:** لا تبدأ البرمجة قبل أن تتخيل كل سيناريوهات الهجوم — استخدم STRIDE.

```
THE STRIDE MODEL (نموذج STRIDE):
=================================

S — Spoofing (انتحال الهوية)
  Can an attacker pretend to be another user?
  Attack: Steal JWT tokens via XSS ? impersonate user
  Defense: HTTPOnly cookies (JS can't read), short token expiry, token rotation

T — Tampering (العبث بالبيانات)
  Can an attacker modify data in transit or at rest?
  Attack: Man-in-the-middle, database injection, parameter tampering
  Defense: HTTPS everywhere, input validation, HMAC signatures on sensitive data

R — Repudiation (الإنكار)
  Can a user deny they performed an action?
  Attack: "I never made that payment" / "I never deleted that data"
  Defense: Comprehensive audit logs, digital signatures, timestamps

I — Information Disclosure (تسريب المعلومات)
  Can an attacker read data they shouldn't?
  Attack: SQL injection, path traversal, over-fetched API responses
  Defense: Principle of least privilege, parameterized queries, field-level authorization

D — Denial of Service (حجب الخدمة)
  Can an attacker make the system unavailable?
  Attack: DDoS, account enumeration, resource exhaustion
  Defense: Rate limiting, CAPTCHAs, CloudFlare DDoS protection, circuit breakers

E — Elevation of Privilege (رفع الصلاحيات)
  Can a regular user gain admin powers?
  Attack: Insecure direct object reference (IDOR), privilege escalation via API params
  Defense: Role-based access control (RBAC), always check permissions server-side
```

### D2: The OWASP Top 10 Implementation Guide (دليل تطبيق معايير OWASP)

```
OWASP TOP 10 — 2023 WITH REAL IMPLEMENTATIONS
-----------------------------------------

1. BROKEN ACCESS CONTROL (فشل التحكم بالوصول)
   -----------------------------------------
   Attack: GET /api/users/12345/documents ? returns documents
           Attacker changes to GET /api/users/99999/documents ? ALSO works
   
   This is called IDOR (Insecure Direct Object Reference)
   
   Fix — EVERY API endpoint must check: "Does this user own this resource?"
   
   ```typescript
   // WRONG:
   app.get('/api/documents/:id', async (req, res) => {
     const doc = await Document.findById(req.params.id); // No ownership check!
     res.json(doc);
   });
   
   // RIGHT:
   app.get('/api/documents/:id', authenticate, async (req, res) => {
     const doc = await Document.findOne({
       id: req.params.id,
       userId: req.user.id  // MUST match the requesting user
     });
     if (!doc) return res.status(404).json({ error: 'Not found' });
     res.json(doc);
   });
   ```

2. CRYPTOGRAPHIC FAILURES (فشل التشفير)
   -----------------------------------------
   Attack: Database leaked ? passwords in plaintext or MD5 ? all accounts compromised
   
   Fix: 
   ```typescript
   // NEVER store plaintext passwords
   // NEVER use MD5 or SHA1 for passwords
   // ALWAYS use bcrypt, Argon2, or scrypt
   
   import bcrypt from 'bcrypt';
   
   // Storing:
   const hashedPassword = await bcrypt.hash(plainPassword, 12); // cost factor 12
   
   // Verifying:
   const isValid = await bcrypt.compare(plainPassword, hashedPassword);
   ```
   
   Argon2 is MORE secure than bcrypt (winner of Password Hashing Competition):
   ```typescript
   import argon2 from 'argon2';
   const hash = await argon2.hash(password);
   const valid = await argon2.verify(hash, password);
   ```

3. INJECTION (الثغرات الحقنية)
   -----------------------------------------
   SQL Injection — most dangerous: 
   ```sql
   -- ATTACK: email = "user@evil.com'; DROP TABLE users;--"
   -- WRONG (vulnerable):
   SELECT * FROM users WHERE email = '${email}'
   
   -- RIGHT (parameterized):
   SELECT * FROM users WHERE email = $1  -- with [email] as parameter
   ```
   
   With ORMs (Prisma, TypeORM, Sequelize): Safe by default
   With raw SQL: ALWAYS use parameterized queries

4. INSECURE DESIGN (التصميم غير الآمن)
   -----------------------------------------
   This is about security in the DESIGN phase, not just implementation.
   
   Example: A "Forgot Password" flow that sends the password instead of a reset link
   ? The design itself is insecure
   
   Secure Password Reset Design:
   1. User requests reset
   2. Generate cryptographically random token (crypto.randomBytes(32))
   3. Store hash of token in DB with 1-hour expiry
   4. Email link: https://yourapp.com/reset?token=RAW_TOKEN
   5. User clicks link ? verify raw token against stored hash
   6. If valid ? allow new password ? INVALIDATE token immediately

5. SECURITY MISCONFIGURATION (سوء الإعدادات الأمنية)
   -----------------------------------------
   Most common misconfigurations:
   
   ? Default credentials (admin/admin, admin/password)
   ? Stack traces exposed in production responses
   ? Debug mode enabled in production
   ? Unnecessary features/services enabled
   ? Missing security headers
   
   Required HTTP Security Headers:
   ```
   Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
   X-Content-Type-Options: nosniff
   X-Frame-Options: DENY
   Content-Security-Policy: default-src 'self'; ...
   Referrer-Policy: strict-origin-when-cross-origin
   Permissions-Policy: geolocation=(), camera=(), microphone=()
   ```
   
   Use: helmet.js (Node.js) for automatic headers

6. VULNERABLE AND OUTDATED COMPONENTS (المكونات الضعيفة والقديمة)
   -----------------------------------------
   Tool: npm audit (check for CVEs in dependencies)
   Automation: Dependabot (GitHub) ? auto PRs for security updates
   Tool: Snyk ? continuous security monitoring
   
   RULE: Run `npm audit` in CI/CD. Fail build if HIGH or CRITICAL CVEs found.

7. IDENTIFICATION AND AUTHENTICATION FAILURES (فشل التعريف والمصادقة)
   -----------------------------------------
   Brute Force Protection:
   ```typescript
   // Rate limit login attempts
   const loginLimiter = rateLimit({
     windowMs: 15 * 60 * 1000, // 15 minutes
     max: 10, // 10 attempts per window
     skipSuccessfulRequests: true,
     message: 'Too many login attempts'
   });
   app.post('/login', loginLimiter, loginHandler);
   ```
   
   Account Enumeration Protection:
   // WRONG: "No user found with this email" ? attacker knows email doesn't exist
   // RIGHT: "If this email exists, you'll receive a reset link" (same message either way)

8. SOFTWARE AND DATA INTEGRITY FAILURES (فشل سلامة البرمجيات والبيانات)
   -----------------------------------------
   Supply Chain Attack (SolarWinds was this):
   - Malicious code injected into npm packages
   - Defense: Use exact version pinning (package-lock.json) + npm audit
   - Use: Socket.dev for supply chain security
   
   Subresource Integrity (SRI) for CDN resources:
   ```html
   <script src="https://cdn.example.com/lib.js" 
           integrity="sha384-[hash]" 
           crossorigin="anonymous"></script>
   ```

9. SECURITY LOGGING AND MONITORING FAILURES (فشل السجلات والمراقبة الأمنية)
   -----------------------------------------
   Log these events ALWAYS:
   - All authentication attempts (success + failure)
   - All authorization failures
   - All password change events
   - All admin actions
   - All data exports
   - All payment events
   
   Alert on:
   - 5+ failed login attempts from same IP ? Brute force
   - Login from new country ? Possible account takeover
   - Unusual data export volume ? Possible data theft

10. SERVER-SIDE REQUEST FORGERY (SSRF)
    -----------------------------------------
    Attack: API accepts a URL ? attacker sends internal network URL
    "Scan this website: http://169.254.169.254/latest/meta-data/" ? 
    Returns AWS instance metadata including credentials!
    
    Defense:
    - Whitelist allowed domains
    - Block private IP ranges (10.x.x.x, 172.16.x.x, 192.168.x.x)
    - Block link-local (169.254.x.x)
    - Use allowlist-based URL validation
```

```
THE GOLDEN RULE: No secrets in code. Ever. Period.
-----------------------------------------

HIERARCHY OF SECRETS MANAGEMENT:

### D4: The Zero-Trust Architecture (عمارة الثقة المعدومة)

```
ZERO-TRUST PRINCIPLES:
=================================

"Never trust, always verify" — even inside your own network

Old model: "If you're inside the firewall, you're trusted"
  ? Problem: Once attacker is inside (e.g., compromised server), they can move freely

Zero-Trust model: EVERY request must be verified, regardless of origin

THE 5 PILLARS:

1. IDENTITY VERIFICATION
   Every service call must include authentication
   Service-to-service: Use mutual TLS (mTLS) or service tokens
   No "internal only" endpoints without authentication

2. LEAST PRIVILEGE ACCESS
   Every service only has access to what it NEEDS
   Database user for the API server: Can only SELECT/INSERT on specific tables
   NOT: Full superuser access "for convenience"
   
   Implementation:
   ```sql
   CREATE USER api_service WITH PASSWORD '...';
   GRANT SELECT, INSERT, UPDATE ON users, sessions TO api_service;
   GRANT SELECT ON products, categories TO api_service;
   -- api_service CANNOT drop tables, access admin tables, etc.
   ```

3. ASSUME BREACH
   Design the system as if an attacker has already penetrated one component
   "If our API server is compromised, what can the attacker do?"
   Answer should be: "Only access the specific data that server needs"
   NOT: "Access the entire database and all other services"

4. MICRO-SEGMENTATION
   Network segments isolate services from each other
   Database server: ONLY accepts connections from app servers
   NOT: Accessible from the public internet
   
   In Kubernetes: NetworkPolicy resources
   In AWS: Security groups with explicit allow rules

5. CONTINUOUS MONITORING
   Assume nothing is normal until proven normal
   Every API call is logged and analyzed
   Anomaly detection: "This service never queries this table at 3 AM"
   Tools: AWS CloudTrail, Datadog APM, Elastic SIEM
```

### E1: Frontend Performance Bible (دستور أداء الواجهات الأمامية)

```
CORE WEB VITALS — Google's Performance Standard
-----------------------------------------

1. LCP — Largest Contentful Paint (سرعة عرض المحتوى الأساسي)
   Measures: How fast the main content appears
   GOOD: < 2.5s
   NEEDS IMPROVEMENT: 2.5-4.0s
   POOR: > 4.0s
   
   What causes slow LCP:
   - Large unoptimized images (most common)
   - Render-blocking JavaScript
   - Slow server response
   - No preloading of hero images
   
   Fixes:
   ```html
   <!-- Preload the LCP image -->
   <link rel="preload" as="image" href="/hero.webp" fetchpriority="high">
   
   <!-- Use modern image formats -->
   <picture>
     <source type="image/avif" srcset="hero.avif">
     <source type="image/webp" srcset="hero.webp">
     <img src="hero.jpg" alt="Hero" width="1200" height="600" loading="eager">
   </picture>
   ```
   
   For Next.js:
   ```tsx
   <Image 
     src="/hero.jpg" 
     priority // preloads the image
     sizes="100vw"
     width={1200}
     height={600}
   />
   ```

2. INP — Interaction to Next Paint (سرعة الاستجابة للتفاعل)
   Replaced FID in 2024. Measures responsiveness to interactions.
   GOOD: < 200ms
   NEEDS IMPROVEMENT: 200-500ms
   POOR: > 500ms
   
   Causes of poor INP:
   - Long JavaScript tasks blocking the main thread
   - Expensive event handlers (sorting, filtering 10,000 items on click)
   - Layout thrashing (reading then writing DOM repeatedly)
   
   Fixes:
   - Break long tasks into smaller chunks with scheduler API
   - Use Web Workers for CPU-intensive work
   - Debounce expensive operations
   - Use React.useMemo/useCallback for expensive computations
   - Virtual scrolling for long lists (react-virtual, TanStack Virtual)

3. CLS — Cumulative Layout Shift (ثبات العناصر البصرية)
   Measures: How much the page jumps around while loading
   GOOD: < 0.1
   NEEDS IMPROVEMENT: 0.1-0.25
   POOR: > 0.25
   
   Causes:
   - Images without dimensions (browser doesn't know size before load)
   - Ads/embeds without reserved space
   - Dynamic content injected above existing content
   
   Fixes:
   - ALWAYS specify width/height on img tags
   - Reserve space for ads with min-height
   - Use CSS aspect-ratio for responsive elements

PERFORMANCE OPTIMIZATION HIERARCHY (سلم الأولويات):
-------------------------------------------------

LEVEL 1 — NETWORK (biggest gains):
  [ ] Enable compression (Brotli > gzip, always prefer Brotli)
  [ ] Use CDN (Cloudflare, Vercel Edge, AWS CloudFront)
  [ ] Enable HTTP/3 (Cloudflare does this automatically)
  [ ] Minimize HTTP requests (bundle JS/CSS)
  [ ] Preconnect to critical third-party domains
      <link rel="preconnect" href="https://fonts.googleapis.com">

LEVEL 2 — IMAGES (second biggest gains):
  [ ] Convert to WebP/AVIF (70% smaller than JPEG)
  [ ] Implement responsive images with srcset
  [ ] Lazy load below-the-fold images (loading="lazy")
  [ ] Use image CDN (Cloudflare Images, imgix, Cloudinary)
  [ ] Compress without visible quality loss (squoosh.app)

LEVEL 3 — JAVASCRIPT (runtime performance):
  [ ] Code splitting (dynamic imports for non-critical features)
  [ ] Tree shaking (eliminate dead code)
  [ ] Analyze bundle: npx @next/bundle-analyzer
  [ ] Replace heavy libraries with lighter alternatives:
      moment.js (16KB) ? date-fns (cherry-picked) or Temporal API
      lodash (70KB full) ? specific lodash functions or native methods
      jQuery ? vanilla JS (it's 2026, jQuery has no place)

LEVEL 4 — FONTS (often overlooked):
  [ ] font-display: swap (show system font first, swap when loaded)
  [ ] Subset fonts (only include characters you need)
  [ ] Host fonts locally (avoid external font requests)
  [ ] Prefer system fonts for body text (fastest: no network request)

LEVEL 5 — RENDERING:

### E2: Backend Performance Bible (دستور أداء الأنظمة الخلفية)

```
DATABASE QUERY OPTIMIZATION
-----------------------------------------

THE N+1 QUERY PROBLEM (كارثة الاستعلامات المتعددة):
-------------------------------------------------
// WRONG — N+1 queries (1 + N users):
const users = await User.findAll();
for (const user of users) {
  user.posts = await Post.findAll({ where: { userId: user.id }});
  // This runs 1 query per user ? 1001 queries for 1000 users!
}

// RIGHT — 2 queries with join:
const users = await User.findAll({
  include: [{ model: Post }]  // One JOIN query
});

// Or using Prisma:
const users = await prisma.user.findMany({
  include: { posts: true }
});

EXPLAIN ANALYZE — Your Best Friend:
  Run EXPLAIN ANALYZE on any slow query.
  Look for:
    - Seq Scan on large tables ? Missing index!
    - Nested Loop with large row count ? Consider restructuring
    - Sort on non-indexed column ? Add index
    - Filter after load ? Move filter into WHERE clause

INDEX STRATEGY:
  Index on: Columns you filter by (WHERE), join on, or sort by (ORDER BY)
  
  Composite indexes — ORDER MATTERS:
  Index on (user_id, created_at) helps:
    WHERE user_id = 5 AND created_at > '2024-01-01' ?
    WHERE user_id = 5 ? (leftmost prefix works)
    WHERE created_at > '2024-01-01' ? (partial, worse)
    
  Partial indexes for common filters:
  CREATE INDEX idx_active_users ON users(email) WHERE active = true;
  ? Much smaller index, much faster for common "active user" queries

PAGINATION BEST PRACTICES:
  WRONG (offset pagination gets slow at scale):
  SELECT * FROM posts ORDER BY created_at DESC LIMIT 20 OFFSET 10000
  ? DB must read 10020 rows and discard 10000 ? O(offset)
  
  RIGHT (cursor-based pagination):
  SELECT * FROM posts 
  WHERE created_at < :last_cursor 
  ORDER BY created_at DESC LIMIT 20
  ? Always fast regardless of page number ? O(1)

CONNECTION POOLING:
  Without pooling: Each request opens/closes a DB connection (expensive)
  With pooling: Connections are reused
  
  Node.js: pg-pool (PostgreSQL), Prisma has built-in pooling
  Production: Use PgBouncer (connection pooler for PostgreSQL)
  
  Optimal pool size ˜ Number of CPU cores × 2 + effective_spindle_count
  For most apps: Start with pool_size = 10-20

CACHING STRATEGY (بروتوكول التخزين المؤقت):
-------------------------------------------------
LEVEL 1 — In-Memory Cache (fastest, not shared):
  Cache in the process memory using a Map or LRU cache
  Best for: Expensive computations that are user-agnostic
  Library: lru-cache (Node.js)
  
LEVEL 2 — Redis (shared, persistent):
  Cache in Redis for data shared across multiple server instances
  Best for: User sessions, frequently-read data, rate limiting
  Patterns:
  
  Cache-aside (most common):
  ```typescript
  async function getUser(id: string) {
    // Check cache first
    const cached = await redis.get(`user:${id}`);
    if (cached) return JSON.parse(cached);
    
    // Cache miss: fetch from DB
    const user = await db.user.findById(id);
    
    // Store in cache (expire in 5 minutes)
    await redis.setex(`user:${id}`, 300, JSON.stringify(user));
    
    return user;
  }
  ```
  
  Cache invalidation:
  ```typescript
  async function updateUser(id: string, data: Partial<User>) {
    await db.user.update(id, data);
    await redis.del(`user:${id}`); // Invalidate cache
  }
  ```

LEVEL 3 — HTTP Caching (CDN layer):
  Cache static content at edge, closest to user
  Headers:
  ```
  Cache-Control: public, max-age=31536000, immutable  (for static assets)
  Cache-Control: private, no-cache  (for authenticated content)
  Cache-Control: public, s-maxage=60  (for public API responses, CDN caches 60s)
  ```

QUERY RESULT CACHING:
  Redis for expensive computed results:
  - "Top 10 trending products" ? compute once, cache 5 minutes
  - "User's permission set" ? cache until permissions change
  - "Dashboard aggregations" ? cache 1 minute
  NEVER cache: Financial calculations, real-time inventory, auth tokens

API RESPONSE OPTIMIZATION:
-------------------------------------------------
Field Selection (GraphQL or REST):
  Only return fields the client needs
  Problem: "Give me all users" returns 40 fields per user ? 95% unused
  Solution: ?fields=id,name,email in REST, or explicit selection in GraphQL

Compression:
  Enable gzip/Brotli for API responses
  For Node.js: compression() middleware (add ~2 lines of code, 70% smaller responses)

HTTP/2 Multiplexing:
  Multiple requests over a single connection
  Enabled by default on most modern platforms
  Eliminates the "6 connection per domain" limit of HTTP/1.1
```

STEP 1: WHAT IS YOUR PRIMARY DATA RELATIONSHIP?

### F1: The CI/CD Pipeline Architecture (هندسة خطوط الإنتاج والتدفق)

```
THE GOLD STANDARD CI/CD PIPELINE
-----------------------------------------

PIPELINE STAGES (in order):
-----------------------------------------
Stage 1: CODE QUALITY (< 2 minutes)
  +-- Linting (ESLint, Prettier) — style and syntax
  +-- Type checking (tsc --noEmit for TypeScript)
  +-- Spell checking (optional but useful for docs)
  +-- Secret scanning (gitleaks) — BLOCKS commit if secrets found

Stage 2: SECURITY (< 5 minutes)
  +-- Dependency audit (npm audit --audit-level=high)
  +-- SAST — Static Application Security Testing (CodeQL, Semgrep)
  +-- License checking (ensure no GPL contamination in commercial code)

Stage 3: TESTING (< 10 minutes)
  +-- Unit tests (Jest, Vitest) — fast, isolated
  +-- Integration tests (with real DB in Docker) — medium
  +-- Coverage threshold check (fail if < 80%)
  +-- Snapshot tests (for UI components)

Stage 4: BUILD (< 5 minutes)
  +-- Build artifacts (npm run build)
  +-- Docker image build
  +-- Image scanning (Trivy, Snyk Container)
  +-- Push to container registry

Stage 5: DEPLOY TO STAGING (< 3 minutes)
  +-- Apply DB migrations (automated)
  +-- Deploy to staging environment
  +-- Run smoke tests (critical path only)
  +-- E2E tests (Playwright against staging)

Stage 6: DEPLOY TO PRODUCTION (manual approval OR automatic)
  +-- Canary deployment (5% traffic first)
  +-- Monitor error rate and latency for 10 minutes
  +-- If healthy: 100% traffic switch
  +-- If unhealthy: Automatic rollback

GITHUB ACTIONS TEMPLATE:
-----------------------------------------
```yaml

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run type-check
      - name: Secret Scanning
        uses: gitleaks/gitleaks-action@v2
        
  security:
    needs: quality
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm audit --audit-level=high
      - uses: github/codeql-action/analyze@v3
      
  test:
    needs: quality
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: test
          POSTGRES_DB: test_db
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test -- --coverage
      - name: Coverage Report
        uses: codecov/codecov-action@v4
        
  build:
    needs: [security, test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ${{ env.REGISTRY }}/app:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
          
  deploy-staging:
    needs: build
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - name: Deploy to staging
        run: ./scripts/deploy.sh staging ${{ github.sha }}
        
  deploy-production:
    needs: build
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production  # Requires manual approval

      - name: Deploy to production
        run: ./scripts/deploy.sh production ${{ github.sha }}
```

DEPLOYMENT STRATEGIES:
-----------------------------------------
1. RECREATE: Stop old, start new. Simple. DOWNTIME.
   Use for: Development environments only. Never production.

2. ROLLING UPDATE: Replace instances gradually.
   Old instances serve traffic while new ones spin up.
   Use for: Most stateless services.
   Risk: Brief period with mixed old/new versions.

3. BLUE/GREEN: Two identical environments. Switch DNS/load balancer.
   Full old version (Blue) running while new (Green) is deployed.
   Use for: Zero-downtime, easy rollback.
   Cost: 2x infrastructure during deployment.

4. CANARY: Route small % of traffic to new version.
   1% ? 5% ? 25% ? 100% with health checks at each stage.
   Use for: High-risk changes to critical services.
   Tool: AWS CodeDeploy, Flagger + Kubernetes, Vercel (automatic).

5. FEATURE FLAGS: Deploy code without activating features.
   Decouple deployment from feature release.
   Use for: Complex features, A/B testing, gradual rollout.
   Tools: LaunchDarkly, Statsig, GrowthBook (open-source), Unleash
```

### F3: Cloud Cost Optimization Protocol (بروتوكول تحسين تكاليف السحاب)

```
CLOUD COST ENGINEERING — SAVE 40-70% ON CLOUD BILLS
-----------------------------------------

THE BIGGEST CLOUD COST MISTAKES:
-----------------------------------------
1. OVERSIZED INSTANCES
   Problem: You provision a c5.4xlarge because "we might need it"
   Reality: CPU average usage is 8%
   Solution: Start small, use CloudWatch to monitor, resize when needed
   Savings: 50-80% reduction possible

2. UNUSED RESOURCES
   Problem: Dev/staging instances running 24/7 when only used 8hrs/day
   Solution: Auto-stop non-production resources during off-hours
   Tool: AWS Instance Scheduler
   Savings: 65% on dev/staging

3. DATA TRANSFER COSTS (the hidden killer)
   Problem: AWS charges for data LEAVING their network
   $0.09/GB egress. 1TB/month = $90/month
   Solution: 
     - CloudFront CDN (covered by CloudFront pricing, much cheaper)
     - Keep services in same region/AZ (same-AZ transfer is free)
     - Compress responses (less bytes = less cost)

4. RESERVED INSTANCES vs ON-DEMAND
   On-demand: Full price, flexible
   Reserved (1-year commitment): 40% discount
   Reserved (3-year commitment): 60% discount
   Savings Plans: Similar discounts, more flexibility
   
   Rule: If you've run a service for 3+ months consistently ? buy Reserved

5. SPOT INSTANCES FOR BATCH WORKLOADS
   Spot = unused EC2 capacity at 90% discount
   Risk: Can be reclaimed with 2-minute warning
   Use for: Video encoding, ML training, batch data processing
   NOT for: Web servers or databases

THE ARCHITECTURE DECISIONS THAT DESTROY COST:
-----------------------------------------
Serverless vs Always-On:
  Lambda/Cloud Functions:
    ? For: Sporadic, event-driven workloads (< 10M requests/month)
    ? Free tier covers millions of requests
    ? For: Consistent high-traffic APIs (gets expensive fast at scale)
    
  Always-on containers:
    ? For: Consistent traffic, cost predictable at scale
    ? For: Sporadic traffic (you pay for idle time)

S3 vs EBS vs EFS:
  S3: Object storage, $0.023/GB/month ? Use for files, backups, assets
  EBS: Block storage, $0.10/GB/month ? Use for database volumes
  EFS: File system, $0.30/GB/month ? Use only when you need shared filesystem

COST MONITORING STACK:
-----------------------------------------
AWS Cost Explorer ? Built-in, analyze spending patterns
AWS Budgets ? Alert when spending exceeds threshold
Infracost ? Preview cost changes from infrastructure PRs
Finout/Spot.io ? Third-party, more detailed cost intelligence

RESERVED VS SERVERLESS DECISION TREE:
-----------------------------------------
Traffic pattern ? Consistent 24/7? ? Use containers + Reserved Instances
Traffic pattern ? Spiky/Event-driven? ? Use Lambda/Serverless
Traffic pattern ? Growing fast? ? Start serverless, migrate at scale
Team size ? 1-3 engineers? ? Serverless (lower ops burden)
Team size ? 5+ engineers? ? Containers (more control, cheaper at scale)

## 🏰 SECTION H: The Code Architecture Bible (دستور هندسة الكود)

> **الكود هو عمارة السيادة.** ابنِ قلاعاً لا تنهدم، وشفرة لا تنكسر. التقيد بالمبادئ الصارمة ليس "اقتراحاً"، بل هو ما يفرق بين جيوش المحترفين وقطاع الطرق من الهواة. أنا أفرض عليك النظافة المعمارية المطلقة.

### H1: The SOLID Principles in Practice (مبادئ SOLID بالتطبيق)

```
SOLID — NOT THEORY, CONCRETE PATTERNS
-----------------------------------------

S — SINGLE RESPONSIBILITY (المسؤولية الواحدة):
  Each module/class/function does ONE thing.
  
  WRONG:
  ```typescript
  class UserController {
    async register(req, res) {
      // Validates input
      if (!req.body.email) return res.status(400)...;
      // Hashes password
      const hash = await bcrypt.hash(req.body.password, 10);
      // Saves to database
      await db.query('INSERT INTO users...');
      // Sends welcome email
      await sendEmail(req.body.email, 'Welcome!', ...);
      // Returns response
      res.json({ success: true });
    }
  }
  ```
  This function does 5 things ? 5 reasons to change it.
  
  RIGHT:
  ```typescript
  class UserController {
    constructor(
      private userService: UserService,
      private emailService: EmailService
    ) {}
    
    async register(req: Request, res: Response) {
      const user = await this.userService.createUser(req.body);
      await this.emailService.sendWelcome(user.email);
      res.json({ success: true });
    }
  }
  
  class UserService {
    async createUser(data: CreateUserDto): Promise<User> {
      const validated = await this.validate(data);
      const hashed = await this.hashPassword(validated.password);
      return this.userRepository.save({ ...validated, password: hashed });
    }
  }
  ```

O — OPEN/CLOSED (المفتوح/المغلق):
  Open for EXTENSION, closed for MODIFICATION.
  
  WRONG:
  ```typescript
  function calculateDiscount(user: User): number {
    if (user.type === 'premium') return 0.2;
    if (user.type === 'vip') return 0.3;
    if (user.type === 'enterprise') return 0.5;
    // Every new user type requires modifying this function
    return 0;
  }
  ```
  
  RIGHT:
  ```typescript
  interface DiscountStrategy {
    calculate(price: number): number;
  }
  
  class PremiumDiscount implements DiscountStrategy {
    calculate(price: number) { return price * 0.8; }
  }
  
  class VIPDiscount implements DiscountStrategy {
    calculate(price: number) { return price * 0.7; }
  }
  
  // Adding new user type = adding new class, no modification needed
  ```

L — LISKOV SUBSTITUTION (استبدال ليسكوف):
  Subclasses must be substitutable for their parent class.
  
  Classic violation:
  ```typescript
  class Bird {
    fly() { ... }
  }
  
  class Penguin extends Bird {
    fly() { throw new Error("Penguins can't fly!"); } // VIOLATION
  }
  ```
  
  Fix: Use composition over inheritance
  ```typescript
  interface FlyingBird { fly(): void; }
  interface SwimmingBird { swim(): void; }
  
  class Eagle implements FlyingBird { fly() { ... } }
  class Penguin implements SwimmingBird { swim() { ... } }
  ```

I — INTERFACE SEGREGATION (فصل الواجهات):
  Don't force classes to implement interfaces they don't use.
  
  WRONG:
  ```typescript
  interface Animal {
    eat(): void;
    sleep(): void;
    fly(): void; // Not all animals fly!
    swim(): void; // Not all animals swim!
  }
  ```
  
  RIGHT:
  ```typescript
  interface Eater { eat(): void; }
  interface Sleeper { sleep(): void; }
  interface Flyer { fly(): void; }
  interface Swimmer { swim(): void; }
  
  class Duck implements Eater, Sleeper, Flyer, Swimmer { ... }
  class Cat implements Eater, Sleeper { ... }
  ```

D — DEPENDENCY INVERSION (عكس التبعية):
  High-level modules should not depend on low-level modules.
  Both should depend on abstractions.
  
  WRONG:
  ```typescript
  class OrderService {
    private emailer = new GmailEmailer(); // Depends on concrete implementation
    
    async placeOrder(order: Order) {
      await this.processOrder(order);
      await this.emailer.send(order.email, 'Order placed!'); // Tightly coupled
    }
  }
  ```
  
  RIGHT:
  ```typescript
  interface EmailService {
    send(to: string, subject: string, body: string): Promise<void>;
  }
  
  class OrderService {
    constructor(private emailService: EmailService) {} // Depends on abstraction
    
    async placeOrder(order: Order) {
      await this.processOrder(order);
      await this.emailService.send(order.email, 'Order placed!', ...);
    }
  }
  
  // Can swap GmailEmailer, SendGridEmailer, MockEmailer without changing OrderService
  ```
```

### H2: The Error Handling Architecture (هندسة التعامل مع الأخطاء)

```
PRODUCTION-GRADE ERROR HANDLING
-----------------------------------------

ERROR HIERARCHY DESIGN:
-----------------------------------------
```typescript
// Base application error
class AppError extends Error {
  constructor(
    public message: string,
    public code: string,        // Machine-readable: 'USER_NOT_FOUND'
    public statusCode: number,  // HTTP status: 404
    public isOperational = true // true = expected error (user error)
                                // false = unexpected (bug, system failure)
  ) {
    super(message);
    this.name = this.constructor.name;
    Error.captureStackTrace(this, this.constructor);
  }
}

// Domain-specific errors
class NotFoundError extends AppError {
  constructor(resource: string, id: string) {
    super(`${resource} with id ${id} not found`, 'NOT_FOUND', 404);
  }
}

class ValidationError extends AppError {
  constructor(message: string, public fields: Record<string, string>) {
    super(message, 'VALIDATION_ERROR', 400);
  }
}

class UnauthorizedError extends AppError {
  constructor(message = 'Authentication required') {
    super(message, 'UNAUTHORIZED', 401);
  }
}

class ForbiddenError extends AppError {
  constructor(action: string) {
    super(`Not permitted to ${action}`, 'FORBIDDEN', 403);
  }
}

class RateLimitError extends AppError {
  constructor(retryAfterSeconds: number) {
    super('Too many requests', 'RATE_LIMITED', 429);
  }
}
```

GLOBAL ERROR HANDLER:
-----------------------------------------
```typescript
// Express global error handler
app.use((error: Error, req: Request, res: Response, next: NextFunction) => {
  // Log EVERYTHING (with context)
  logger.error({
    error: error.message,
    stack: error.stack,
    code: error instanceof AppError ? error.code : 'UNKNOWN',
    userId: req.user?.id,
    requestId: req.id,
    path: req.path,
    method: req.method,
    userAgent: req.headers['user-agent'],
    ip: req.ip,
  });

  if (error instanceof AppError && error.isOperational) {
    // Known operational error ? tell the client
    return res.status(error.statusCode).json({
      success: false,
      error: {
        code: error.code,
        message: error.message,
        ...(error instanceof ValidationError && { fields: error.fields }),
      }
    });
  }

  // Unknown error ? generic message (don't leak internals)
  // Alert on-call if unknown errors spike!
  return res.status(500).json({
    success: false,
    error: {
      code: 'INTERNAL_ERROR',
      message: 'Something went wrong. Please try again later.'
    }
  });
});
```

ASYNC ERROR HANDLING PATTERN:
-----------------------------------------
```typescript
// Without this helper ? unhandled promise rejections crash Node.js
const asyncHandler = (fn: RequestHandler): RequestHandler =>
  (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next);

// Usage:
app.get('/users/:id', asyncHandler(async (req, res) => {
  const user = await userService.findById(req.params.id);
  if (!user) throw new NotFoundError('User', req.params.id);
  res.json(user);
}));

// Errors automatically forwarded to global error handler
```

RESULT PATTERN (FUNCTIONAL ERROR HANDLING):
-----------------------------------------
```typescript
// Railway-oriented programming style
type Result<T, E = Error> = 
  | { success: true; data: T }
  | { success: false; error: E };

async function findUser(id: string): Promise<Result<User>> {
  try {
    const user = await db.users.findById(id);
    if (!user) return { success: false, error: new NotFoundError('User', id) };
    return { success: true, data: user };
  } catch (err) {
    return { success: false, error: err as Error };
  }
}

// Caller handles both cases explicitly
const result = await findUser(userId);
if (!result.success) {
  // Handle error
} else {
  // Use result.data
}
```

## ☠️ SECTION I: The Anti-Pattern & Failure Library (مكتبة الأنماط السيئة والفشل)

> **الفشل هو المعلم الأقسى، وأنا لا أسمح بتكرار أخطاء الماضي.** النجاح يترك أثراً، وكذلك الفشل الذريع. العقل المدبر يدرس حطام الأنظمة الأخرى ليتجنب مصيرها. أنا آمرك بمسح هذا التاريخ من أذهان المطورين الضعفاء والالتزام بأنماطي الفاخرة فقط.

### I1: Architecture Anti-Patterns (الأنماط المعمارية القاتلة)

```
THE PATTERN GRAVEYARD — Anti-Patterns That Kill Startups
-----------------------------------------

ANTI-PATTERN 1: Premature Microservices (التقسيم المبكر)
---------------------------------------------------------
Symptom: "We'll use microservices from day 1 to be scalable"
Reality: 
  - Your team is 3 people, not 300
  - You don't know where the boundaries are yet
  - You're spending 80% of time on infrastructure, 20% on product
  
Real Cost:
  - 3x slower development velocity
  - Complex distributed tracing needed immediately
  - Each service needs its own deployment, monitoring, docs
  - Transaction management becomes hell (distributed transactions)
  
Netflix, Uber, Amazon all started as monoliths. They extracted services
when they had specific scaling problems, not on day 1.

When to extract a service:
  ? When a specific part needs to scale independently (e.g., image processing)
  ? When teams are large enough to own a service independently
  ? When there's a clear, stable boundary in the domain
  ? When you have >50K requests/second on that specific component
  
Rule: Modular Monolith first. Extract services only when you have evidence you need to.

ANTI-PATTERN 2: The Database God Object (قاعدة البيانات المسيطرة)
---------------------------------------------------------
Symptom: All business logic lives in stored procedures and triggers
  
Problems:
  - Logic scattered between app code and database
  - Testing requires a database
  - Refactoring is nearly impossible
  - Database becomes a bottleneck (can't scale it horizontally)
  - PostgreSQL stored procedures are not version-controlled properly
  
Fix: Move business logic to the application layer.
  Database for: storage, querying, relationships, constraints (FK, UNIQUE)
  Application for: business rules, calculations, transformations

ANTI-PATTERN 3: The Config Catastrophe (كارثة الإعدادات)
---------------------------------------------------------
Symptom: 300 lines of if/else based on environment:
  ```
  if env == 'development': ...
  elif env == 'staging': ...
  elif env == 'production': ...
  elif env == 'production-eu': ...
  ```
  
Problems:
  - Impossible to test all code paths
  - Environments drift (staging doesn't match production)
  - New engineer takes days to understand the config
  
Fix: 
  - Environment-specific values only (URLs, credentials) ? environment variables
  - Business logic never changes per environment
  - Use Twelve-Factor App principles: one codebase, inject config via env

ANTI-PATTERN 4: The Leaky Abstraction
---------------------------------------------------------
Symptom: Your "email service" knows about the user's subscription tier:
  ```typescript
  class EmailService {
    async sendEmail(to: string, userId: string) {
      const user = await User.findById(userId); // DB call inside email service?!
      if (user.tier === 'free') {
        // Wrap in unsubscribe footer
      }
    }
  }
  ```
  
Problems:
  - EmailService now depends on UserService ? circular dependency risks
  - Can't test email sending without database
  - EmailService knows too much
  
Fix: Pass everything the service needs:
  ```typescript
  emailService.sendEmail(to, { includeUnsubscribeLink: user.tier === 'free' });
  ```

ANTI-PATTERN 5: The Magic String Cemetery
---------------------------------------------------------
Symptom: String literals scattered throughout the codebase:
  ```typescript
  if (user.status === 'active') ...
  if (order.status === 'pending') ...
  if (plan === 'premium') ...
  ```
  
Problems:
  - Typo causes silent bug: 'aktive' !== 'active'
  - Rename requires global search/replace
  - No autocomplete, no type checking
  
Fix: TypeScript enums or union types:
  ```typescript
  type UserStatus = 'active' | 'inactive' | 'suspended' | 'deleted';
  const STATUS = {
    ACTIVE: 'active',
    INACTIVE: 'inactive',
  } as const;
  ```

ANTI-PATTERN 6: The Callback Pyramid of Doom
---------------------------------------------------------
The classic 2015-era Node.js mistake that still appears:
  ```javascript
  getUserById(id, function(user) {
    getOrdersByUser(user.id, function(orders) {
      getProductsForOrders(orders, function(products) {
        calculateTotal(products, function(total) {
          sendInvoice(user.email, total, function(result) {
            // We're now 5 levels deep. God help us.
          });
        });
      });
    });
  });
  ```
  
Fix: async/await (it's 2026, there's no excuse):
  ```typescript
  const user = await getUserById(id);
  const orders = await getOrdersByUser(user.id);
  const products = await getProductsForOrders(orders);
  const total = calculateTotal(products);
  await sendInvoice(user.email, total);
  ```

ANTI-PATTERN 7: The Optimistic Distributed Transaction
---------------------------------------------------------
Symptom: Making multiple DB writes without transaction:
  ```typescript
  await deductFromAccount(userId, amount);
  await addToAccount(recipientId, amount);
  // If second call fails ? money disappeared!
  ```
  
Fix: Always wrap related writes in a transaction:
  ```typescript
  await db.transaction(async (trx) => {
    await deductFromAccount(userId, amount, trx);
    await addToAccount(recipientId, amount, trx);
    // Either BOTH succeed or BOTH roll back
  });
  ```

ANTI-PATTERN 8: The Synchronous Email/SMS in Request Lifecycle
---------------------------------------------------------
Symptom:
  ```typescript
  app.post('/register', async (req, res) => {
    const user = await createUser(req.body);
    await sendWelcomeEmail(user.email); // Blocks response for 2-3 seconds!
    await sendSlackNotification(...);   // Another 1 second delay
    res.json(user); // User waited 4+ seconds for registration
  });
  ```
  
Fix: Respond immediately, do async work in background:
  ```typescript
  app.post('/register', async (req, res) => {
    const user = await createUser(req.body);
    res.json(user); // Respond immediately!
    
    // Non-critical work happens asynchronously
    setImmediate(async () => {
      await sendWelcomeEmail(user.email);
      await sendSlackNotification(user);
    });
    
    // Or better: use a job queue (BullMQ)
    await emailQueue.add('welcome', { userId: user.id });
  });
  ```
```

### I2: The Startup Failure Patterns (أنماط فشل الشركات الناشئة)

```
TECHNICAL DECISIONS THAT KILL STARTUPS
-----------------------------------------

1. BUILDING BEFORE VALIDATING
   -----------------------------------------
   Pattern: 6 months of building, then discovering no one wants it
   
   The correct order:
   Month 1: Validate with landing page + waitlist + interviews
   Month 2: Build MVP (simplest possible thing that proves the concept)
   Month 3: Get 10 paying users. If can't ? the problem is wrong
   Month 4+: Build properly based on real feedback
   
   Tech shortcut for validation:
   Day 1: Landing page (Framer/Webflow) with "Join Waitlist"
   Week 1: Manual back-end (Airtable, Notion, spreadsheet)
   Week 2-4: Fake the product for early users before building it

2. OVER-ENGINEERING THE INFRASTRUCTURE BEFORE PRODUCT VALIDATION
   -----------------------------------------
   "Let's set up Kubernetes, Terraform, multi-region deployment before launch"
   You have 0 users. Your biggest cost is developer time, not server costs.
   
   Rule: 
   0-1K users ? Vercel + Supabase + Railway (zero DevOps)
   1K-10K users ? Single VPS + Managed DB (minimal DevOps)
   10K-100K users ? Start introducing proper infrastructure
   100K+ users ? You now have the revenue to hire a DevOps engineer

3. NOT OWNING YOUR CUSTOMER RELATIONSHIP
   -----------------------------------------
   Building on top of a single platform (App Store, Marketplace, SEO)
   and not having a direct relationship with customers.
   
   "We got all our users from the App Store. Apple changed the algorithm. 
   Revenue dropped 80% overnight."
   
   Rule: Always build a direct email list. Email is the only channel you own.
   Minimum: Collect email on signup, even if the product is social.

4. PREMATURE OPTIMIZATION OF THE WRONG METRIC
   -----------------------------------------
   Optimizing: "Server response time is 50ms, let's get to 10ms"
   When the real problem is: "We have 5% conversion rate, should be 30%"
   
   Rule: Optimize what's LIMITING growth. Profile before optimizing.
   The Goldratt principle: Fix the constraint, not the non-constraint.

5. TECHNICAL DEBT BANKRUPTCY
   -----------------------------------------
   Moving so fast in year 1 that the codebase becomes unmaintainable.
   Symptoms: 
   - Every new feature takes 5x longer than the last
   - Engineers are afraid to touch old code
   - Bugs appear when apparently unrelated features change
   - Onboarding new engineers takes months
   
   This is often called the "startup trap" — speed at the cost of quality
   makes you slower over time.
   
   Rule: Reserve 20% of sprint capacity for tech debt reduction.
         Never sprint for > 3 months without a dedicated "cleanup sprint."

6. THE DEPENDENCY TRAP
   -----------------------------------------
   Being entirely dependent on a single vendor:
   - "Our entire product is a thin wrapper around OpenAI" ? OpenAI raises prices, you die
   - "We're entirely on Firebase" ? Google changes pricing, you're trapped
   - "All our auth is through Facebook Login" ? Facebook blocks your app, you lose all users
   
   Rule: 
   - Abstract external dependencies behind interfaces
   - Have a migration path for every critical vendor
   - Own the core value creation; vendor out the commodity
```

### J1: The Viral Loop Architecture (هندسة الحلقات الفيروسية)

```
HOW PRODUCTS GROW VIRALLY
-----------------------------------------

VIRAL COEFFICIENT (K-Factor):
  K = (number of invitations sent per user) × (conversion rate of invitations)
  
  K > 1: Product grows without any marketing spend (viral)
  K = 0.5: For every 2 users, you get 1 new user (sub-viral but helpful)
  K < 0.1: Viral loop exists but barely contributes
  
  To increase K:
    1. Make inviting EASY (one click, beautiful invite flow)
    2. Make the invitation VALUABLE to the recipient
    3. Reduce friction in invitation conversion

TYPES OF VIRAL LOOPS:
-----------------------------------------
1. PRODUCT IS THE INVITATION (strongest):
   Dropbox: "You've shared a file with [Name]. Click to view."
   ? Recipient MUST create account to access file
   ? Creating account shows them the product
   ? They understand the value immediately
   
   Figma: "John is sharing a design with you"
   ? Recipient opens in browser, no install needed
   ? They experience the product before signing up
   
   Calendly: "Schedule time with [Name]"
   ? Every meeting booked introduces Calendly to new people
   ? The product markets itself through use

2. EXPLICIT REFERRAL LOOP:
   Dropbox: "Give 1GB, Get 1GB" — both parties benefit
   This grew Dropbox 3900% in 15 months.
   
   Engineering requirements:
   - Unique referral links per user
   - Attribution tracking (who invited whom)
   - Reward fulfillment automation
   - Abuse prevention (no self-referrals, fraud detection)

3. SOCIAL PROOF LOOP:
   LinkedIn: "X people viewed your profile this week"
   ? Triggers vanity ? User updates profile
   ? Updated profile appears in more feeds
   ? More views ? More notifications ? More engagement

4. STATUS LOOP:
   Gaming leaderboards, Grammarly weekly stats, Spotify Wrapped
   Users share because it reflects positively on them.
   
   Engineering: Public shareable results/achievements with branded visuals.

THE GROWTH FLYWHEEL (عجلة النمو الدوارة):
-----------------------------------------
Not a loop but a flywheel — momentum builds on itself.

Amazon's flywheel:
Lower prices ? More customers ? More sellers
More sellers ? More selection ? More customers
More customers ? More Amazon data ? Better recommendations ? More purchases

Build your flywheel:
What does your product do that:
1. Brings more users?
2. Makes the product better FOR EXISTING users?
3. Creates network effects (more users = more value for all)?

If you can design these 3 ? you have a flywheel.
```

### J2: The Retention Engineering Protocol (بروتوكول هندسة الاحتفاظ)

```
RETENTION IS THE FOUNDATION OF GROWTH
-----------------------------------------

THE RETENTION CURVE:
-----------------------------------------
For a product to be sustainable, the retention curve must flatten.

Unhealthy (churning to 0%):
  100% ? 80% ? 60% ? 40% ? 20% ? 10% ? 5% ? 0%
  = Everyone eventually leaves. Your bucket has a hole.

Healthy (flattens at X%):
  100% ? 80% ? 60% ? 50% ? 45% ? 43% ? 43% ? 43%
  = Retained a stable core of highly engaged users.

To flatten the retention curve:
  1. Find what engaged users DO differently ? make others do the same
  2. Find when users churn ? fix the product experience at that point
  3. Build habits: make the product part of daily/weekly routine

THE AHA MOMENT ENGINEERING:
-----------------------------------------
The "Aha Moment" = the point where a user first understands the product's value.

Facebook: Adding 7 friends in the first 10 days = stayed for years.
Twitter: Following 30 accounts in the first day = retained.
Dropbox: Putting at least 1 file in the Dropbox folder = retained.
Slack: Sending 2000 messages as a team = retained.

These aren't magic numbers — they're correlations with users who "got it."

How to find YOUR aha moment:
  1. Identify your most engaged, long-retained users
  2. Look at what actions they took in the first 7 days
  3. Find the action that CORRELATES with retention (not causes)
  4. Design onboarding to guide every new user to that action FAST

ONBOARDING ENGINEERING:
-----------------------------------------
The first 5 minutes determine if a user stays or churns.

Principles:
  1. ONE THING per screen. Don't overwhelm.
  2. SHOW VALUE before asking for anything.
  3. SKIP EVERYTHING non-essential for day 1.
  4. PERSONALIZE: "What do you want to achieve?" ? tailor experience
  5. QUICK WIN: Give them a success in the first 2 minutes.

Anti-patterns in onboarding:
  ? 10-step form before showing the product
  ? "Please verify your email before you can do anything"
  ? Empty state with no guidance: "You have no items. Create one."
  ? Feature tour that shows 30 features to a new user
  ? Requiring credit card before seeing value

The Empty State is not empty:
  When a user first opens your product and sees nothing:
  ? Don't show a blank screen
  ? Show example content (what it looks like with data)
  ? Show a clear call to action to create their first item
  ? GitHub does this perfectly: "Your repository is empty. Push your code."

CHURN INTERVENTION SYSTEM:
-----------------------------------------
Build automated churn detection:
  
  Signals that predict churn (before the user cancels):
  - Login frequency dropped by >50%
  - Last session was >14 days ago (for daily-use products)
  - Feature usage dropped sharply
  - Support ticket submitted and unresolved
  
  Intervention sequence:
  Day 0 (first signal): In-app message "Need help with X?"
  Day 7 (no improvement): Personalized email from CEO/founder
  Day 14 (still declining): Offer to connect with success team
  Day 21 (about to cancel): Retention offer (discount/pause plan)
  Day 30 (cancelled): Exit survey + win-back sequence starts 60 days later
```

### K1: Internationalization Architecture (هندسة التدويل)

```
BUILDING FOR THE WORLD — NOT JUST ONE MARKET
-----------------------------------------

I18N ARCHITECTURE PRINCIPLES:
-----------------------------------------
Never hardcode text in components:
  
  WRONG:
  ```tsx
  <button>Submit Order</button>
  <p>You have 3 items in your cart.</p>
  ```
  
  RIGHT:
  ```tsx
  <button>{t('checkout.submitOrder')}</button>
  <p>{t('cart.itemCount', { count: 3 })}</p>
  // ? "You have 3 items in your cart." (English)
  // ? "لديك 3 عناصر في سلة التسوق." (Arabic)
  // ? "Vous avez 3 articles dans votre panier." (French)
  ```

TRANSLATION MANAGEMENT:
  Files: public/locales/en.json, public/locales/ar.json
  Library: next-intl (Next.js), i18next (universal)
  Management: Crowdin or Phrase (for teams with translators)

PLURALIZATION (الجمع — Much Harder Than You Think):
  English: 1 item, 2 items (2 forms)
  Arabic: 1 عنصر 2 عنصران 3-10 عناصر 11+ عنصراً (6 forms!)
  Russian: 6 plural forms
  
  ALWAYS use ICU message format for plurals, NEVER string concatenation:
  ```json
  {
    "itemCount": "{count, plural, =0 {No items} =1 {One item} other {# items}}"
  }
  ```

RTL (Right-to-Left) SUPPORT (دعم اللغة العربية من اليمين لليسار):
  Required for: Arabic, Hebrew, Persian, Urdu
  
  CSS logical properties (the right way):
  ```css
  /* WRONG — doesn't flip in RTL: */
  .element { margin-left: 16px; padding-right: 8px; }
  
  /* RIGHT — automatically flips in RTL: */
  .element { margin-inline-start: 16px; padding-inline-end: 8px; }
  ```
  
  HTML attribute: <html dir="rtl" lang="ar">
  
  Common RTL gotchas:
  - Icons with directional meaning (arrows) should flip: scaleX(-1)
  - Progress bars fill from right to left in RTL
  - Animations that slide in from left should slide from right
  - Shadow direction (drop-shadow from left in LTR, from right in RTL)
  - Number formatting: Arabic-Indic digits vs. Western Arabic digits
    In Arabic contexts: "???" vs "342" (both are valid, context matters)

DATE, TIME, AND NUMBER FORMATTING:
  ```typescript
  // ALWAYS use Intl APIs — NEVER manual formatting:
  
  // Date
  new Intl.DateTimeFormat('ar-SA', { 
    dateStyle: 'full' 
  }).format(new Date()) 
  // ? "الإثنين، ٣١ مارس ٢٠٢٦"
  
  // Number
  new Intl.NumberFormat('ar-SA').format(1234567.89)
  // ? "١٬٢٣٤٬٥٦٧٫٨٩"
  
  // Currency
  new Intl.NumberFormat('ar-SA', { 
    style: 'currency', 
    currency: 'SAR' 
  }).format(9.99)
  // ? "٩٫٩٩ ر.س."
  
  // Relative time
  new Intl.RelativeTimeFormat('ar', { 
    numeric: 'auto' 
  }).format(-3, 'day')
  // ? "قبل ٣ أيام"
  ```

TIMEZONE HANDLING:
  NEVER store dates as local time strings.
  ALWAYS store as UTC in the database.
  Convert to user's local timezone only at display time.
  
  ```typescript
  // Storage: always UTC
  const createdAt = new Date().toISOString(); // "2026-03-09T12:00:00.000Z"
  
  // Display: convert to user timezone
  new Intl.DateTimeFormat('en-US', {
    timeZone: userTimezone, // e.g., "Asia/Riyadh"
    dateStyle: 'short',
    timeStyle: 'short'
  }).format(new Date(createdAt));
  ```

## 👁️ SECTION L: The Observability & Monitoring Bible (دستور المراقبة والتحليل)

> **أنا أرى كل شيء.** في مملكتي الرقمية، لا يوجد مكان للاختباء. النظام الذي لا يوفر لي الرؤية الكاملة هو نظام خارج عن السيطرة، وأنا لا أتحمل العشوائية. آمرك ببناء عيون لا تنام (Monitoring) وعقول لا تخطئ (Observability).

### PILLAR 1: LOGS (السجلات)
-----------------------------------------
Structured logging (NOT console.log):

```typescript
// WRONG:
console.log('User created: ' + userId);
console.log('Error: ' + error.message);

// RIGHT (structured JSON):
import pino from 'pino';
const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  base: { service: 'api-server', version: process.env.APP_VERSION },
});

logger.info({ userId, email: user.email, tier: user.tier }, 'User created');
logger.error({ error: error.message, stack: error.stack, requestId }, 'Request failed');
```

Why structured logging:
  ? Can search: "all errors for userId=123"
  ? Can aggregate: "error rate by endpoint"
  ? Can alert: "errors > 10 in 1 minute"
  ? Can trace: follow requestId across services

Log levels (use correctly):
  DEBUG: Detailed debugging info (only in development)
  INFO: Normal operations (user signed up, payment processed)
  WARN: Something unexpected but non-critical (deprecated API used)
  ERROR: Something failed (payment failed, email send failed)
  FATAL: System is unusable (database connection failed permanently)

PILLAR 2: METRICS (المقاييس)
-----------------------------------------
Types of metrics:
  COUNTER: Only goes up (requests served, errors occurred)
  GAUGE: Can go up or down (active users, CPU usage, queue depth)
  HISTOGRAM: Distribution of values (request latency, response sizes)

THE GOLDEN SIGNALS (4 key metrics for any service):
  1. LATENCY: How long do requests take?
     Track: p50, p95, p99 (not just average!)
     Why p99: Average hides the worst experience. 1% of 1M users = 10,000 unhappy users.
     
  2. TRAFFIC: How many requests per second?
     Track: Requests/second, broken down by endpoint and status code
     
  3. ERRORS: What fraction of requests fail?
     Track: Error rate = errors/total requests. Alert if > 1% for 5 minutes.
     
  4. SATURATION: How "full" is the service?
     Track: CPU%, memory%, queue depth, DB connection pool usage
     Alert if > 80% consistently.

SLA/SLO/SLI Framework:
  SLI (Indicator): The actual measurement ("99.2% of requests succeeded in last 24h")
  SLO (Objective): Your internal target ("99.5% availability per month")
  SLA (Agreement): The contractual commitment to customers ("99.9% availability")
  
  Error Budget: SLO - Actual availability = Budget remaining to "spend" on risk
  If SLA is 99.9% ? you have 43 minutes of downtime budget per month.
  Spend it wisely.

PILLAR 3: TRACES (التتبع)
-----------------------------------------
Distributed traces show the full path of a request across services.

When a request touches: API ? Auth Service ? Database ? Cache ? External API
? A trace shows EXACTLY which service took how long
? Spans show nested operations within a service

Tools:
  Jaeger (open-source)
  Zipkin (open-source)
  Datadog APM (commercial, excellent)
  Honeycomb (best for high-cardinality tracing)

Implementation (OpenTelemetry — the standard):
```typescript
import { trace } from '@opentelemetry/api';

const tracer = trace.getTracer('api-server');

async function getUserOrders(userId: string) {
  const span = tracer.startSpan('getUserOrders');
  span.setAttribute('user.id', userId);
  
  try {
    const orders = await db.orders.findMany({ where: { userId } });
    span.setAttribute('orders.count', orders.length);
    return orders;
  } catch (error) {
    span.setStatus({ code: SpanStatusCode.ERROR, message: error.message });
    throw error;
  } finally {
    span.end();
  }
}
```

-----------------------------------------
Budget ($0-100/month):

  Metrics: Grafana Cloud (free tier)

-----------------------------------------

  Logs + Metrics + Traces: Datadog (all-in-one, best DX)
  Alternative: New Relic (similar, slightly cheaper)
  Uptime + Alerts: PagerDuty or OpsGenie

  Self-hosted ELK Stack (Elasticsearch + Logstash + Kibana)

      For scale-ups ? Datadog (operational simplicity > cost savings).

ALERTING ARCHITECTURE:
-----------------------------------------
Alert fatigue is real. Bad alerts = alerts ignored = incidents missed.

THE ALERT HIERARCHY:
  P1 — CRITICAL (wake someone up at 3am):
    ? Production is DOWN (error rate > 50%)
    ? Data loss is happening
    ? Security breach detected
    ? Payment processing failed completely
    
  P2 — HIGH (alert during business hours):

    ? P99 latency > 5 seconds
    ? Database connection pool > 90%

  P3 — MEDIUM (daily digest):

## 🎲 SECTION M: The Decision-Making Under Uncertainty Protocol

القرار الصحيح في الوقت الخاطئ هو قرار خاطئ — تعلم كيف تقرر.

M1: The Uncertainty Quantification Framework
HOW TO DECIDE WHEN YOU DON'T HAVE ENOUGH DATA
-----------------------------------------

THE 3 TYPES OF UNKNOWNS (أنواع المجاهيل في اتخاذ القرارات):

1. KNOWN KNOWNS: Things you know that you know
   ? We know PostgreSQL has ACID compliance
   ? We know our current traffic is 10K req/day
   ? Action: Use this confidently in your decision

2. KNOWN UNKNOWNS: Things you know that you don't know
   ? We don't know how many users we'll have in 6 months
   ? We don't know if users will prefer feature A or B
   ? Action: Design to be flexible in these dimensions. Run experiments.

3. UNKNOWN UNKNOWNS: Things you don't know that you don't know
   ? "We had no idea our users would use the product this way"
   ? "We didn't know this regulation would apply to us"
   ? Action: Build reversibility into all major decisions. Stay humble.

THE PRE-MORTEM PROTOCOL (تحليل الفشل الاستباقي):
-----------------------------------------
Before starting any major project:
Gather the team. Say: "It's one year from now. The project FAILED.
Tell me every possible reason why."

This surfaces:
  ? Risks people were afraid to say out loud
  ? Blind spots in the plan
  ? Dependencies no one thought to check
  ? Resource constraints that were glossed over

Then for each risk identified:
  Probability: Low / Medium / High
  Impact: Low / Medium / High
  Mitigation: What do we do to prevent or reduce it?
  
  HIGH PROBABILITY + HIGH IMPACT ? Redesign the plan. This is not acceptable risk.
  LOW PROBABILITY + HIGH IMPACT ? Have a response plan ready (runbook).
  HIGH PROBABILITY + LOW IMPACT ? Accept it. Not worth the mitigation cost.
